Buffer Lanes

Updated 23.02.2024

Introduction

Buffer lanes are a feature that allows unloading resources one after another and similarly loading first resource from a line of resources. Normally there can be only one resource per symbolic point but with this feature it is possible to add a buffer area, that can fit multiple resources. Figure 1 shows example of the unload buffer. On that Figure, the object is detected with scanner and then the AGV does the unload in specified distance from that detected object.

Figure 1: Example picture of unload buffer.

Unload buffer

Unload buffer is a feature that allows AGV to unload resources in a line. When AGV is inside the buffer area, it uses laser scanner to detect where the last unloaded resource is and calculates the unload location for the carried resource. Then AGV drives to that calculated unload location, performs unload and continues driving to next mission.

Load buffer

Load buffer is a feature that allows AGV to load the last resource from a line of resources. When AGV is inside the buffer area, it uses sensor information to detect where the last resource is on the line and calculates load location to be able to pick the resource. There are two sensor options: 1. Laser scanner to detect where the last resource is on the buffer area or 2. Load sensor information to know when the forks are under a resource to pick it up. Then AGV drives to the calculated load location, performs load and continues to unload location.

Requirements

  • Pallet detection license

  • Navitrol 6.28 (6.32 for docking)

  • Laser scanner to detect the object on the buffer area.

  • Navithor 2.3.0.2

Buffer lane feature

This Chapter explains buffer lane load and unload features with more detail and Navitrol parameters that affects to the performance.

Unload buffer

Laser scanner for object detection:

  • Must see the resource when AGV is driving on buffer area

  • Must see a cluster of scanner measurement points

  • Full pallet detection is not needed

Cluster parameters:

  • pallet_det_min_points_close_to_first_point defines how many scanner points are required for cluster

  • pallet_det_2_max_dist_to_first_point defines distance between consecutive cluster points.

AGV distance to detected object:

  • buffer_stop_dist_from_object_when_unloading parameter defines the distance from detected object where to perform the unload.

Search area:

  • pallet_model_width_y[0] parameter defines the search area, where the scanner points are searched for. See Figure 2 below.

Figure 2: Unload search area

Load buffer

Load buffer has three different modes:

  1. Object detected with scanner

  2. Pallet detected with scanner

  3. Load sensor is triggered.

Figure 3: Load buffer search area

Mode 1: Object detection with scanner

This is the same functionality as on the unload. This means that the scanner needs only few scanner points close to each other to detect an object on the route buffer (See above what parameters affects to the clusters).

  • buffer_stop_dist_from_object_when_loading defines the stopping point from the detected object and the distance is calculated from navigation point to the object.

The downside with this mode is that it might detect anything on the route, so the accuracy might vary.

Mode 2: Pallet detection with scanner

This mode uses full pallet detection and also has this object detection feature (same as mode 1) as a safety.

  • buffer_stop_dist_from_object_when_loading defines the stopping point and the distance is calculated from navigation point to the pallet's front line.

  • buffer_stop_distance_from_unwanted_object defines the distance from detected object to stop the AGV if pallet is not detected.

  • buffer_max_dist_from_route_to_target defines maximum allowed sideways distance that the pallet can have from the route. Error is set if the pallet is further.

Mode 3: Load sensor

Load sensor needs to give Navitrol information when the AGV has driven under the pallet and is ready for performing the load. The AGV is driving on the route and waiting this load sensor to trigger, otherwise it will just drive to the end symbolic point.

  • buffer_stop_dist_from_object_when_loading defines the distance how far the AGV can still drive after the load sensor is triggered. In many cases, this can be small number e.g., 0.05, the sign depends on the driving direction.

Docking

Docking is released on 6.32 release. It means that AGV can correct its position sideways on the buffer area. This Chapter explains only the basic idea of Docking. More info can be found from Navitrol Pallet Detection Instructions document

These parameters have to be set to enable docking:

  • docking_to_target_enabled 1 (omni or mechanum) or 2 (steer or differential drive AGV)

  • Load_buffer_mode 2

  • Docking_stop_distance replaces the buffer_stop_dist_from_object_when_loading when docking is used.

NOTE! The buffer_check_start_dist_offset_m should have large enough value to allow some space for the docking. If this parameter is zero or too small, the pallet detection starts when the AGV is inside the buffer area. This can be too late to start docking if the buffer is full and the pallet is very close to the buffer start.

Docking requires pallet detection tag to be enabled on Tools, otherwise the AGV will drive only on the route as in normal buffer lane.

Figure 4: Docking requires pallet detection tag for the Symbolic point on Tools.

The docking has two extra start conditions for load buffer. See Figure 5 and Figure 6.

  • The calculated stop point has to be closer to the route than the buffer_max_dist_from_route_to_target parameter.

  • The calculated stop point cannot be further away from the Symbolic point (behind the SP) than the docking_max_dist_error_from_end_point parameter. This replaces the normal docking start condition that check only the target distance to the SP.

Figure 5: First start condition: Distance to route cannot be too high. It is calculated sideways (see the red arrow).

Figure 6: Second start condition: Stop point cannot be too far behind the Symbolic point.

Other start conditions and more information about the docking can be found from the Navitrol Pallet Detection Instructions document.

Creating Buffer Lanes

Creating a buffer lane for a Load or Unload point is done by setting a buffer distance and enabling the buffer. This can only be done for symbolic points with the type Load or Unload. The size of the lane is visualized in magenta. The UI is displayed in Figure 5.

As a limitation buffer lane cannot currently span across multiple links. Additionally, the load or unload point must only have a single link connected thus it needs to be at the end of a link.

NOTE: buffer_check_start_dist_offset_m, 2.0 by default**,** parameter is offset value that is added to the buffer lane start point. If the buffer lane is 10m on the Tools and this parameter is 1.0 m, it means that the buffer is actually 11.0 m long.

Figure 7. Creating Buffer Lanes. 1: Symbolic point type has to be Load or Unload or both. 2: Enable Buffer Lane feature. 3: Set Buffer area distance, illustrated in magenta on the route.

Fleet control settings

Using buffer buffer lanes in production requires following TCPara­meters to be set:

  • ResourceManager_ResourceContainerType -- Single

  • ResourceManager_ConstainerStackType -- CombineSameType

Optionally:

  • ProductionManager_UnloadLocationAssignMode -- buffer_balance

  • SymbolicPoint_BUFFER_FULL_DISTANCE_REMAINING -- any value

Unload location assign mode buffer_balance prioritizes unload point with most space left in the buffer. Navithor updates the space remaining in a buffer every time a resource is unloaded to it. The buffer is full if the symbolic point resource count is full or if there is no space on the buffer area to unload anymore pallets. The resource length is taken from the machine type parameter xml file. If the point is set empty, the space left in buffer is also reset.

Buffer full distance remaining sets when we consider the buffer to be full. The buffer is considered full when Navithor believes that the space left in buffer is lower than this value.

Grouping symbolic points

If there are multiple buffer lanes on the production area, the symbolic points can be grouped. Then the production orders can be set to the whole group and Navithor chooses the target buffer based on the UnloadLocationAssignMode parameter. When one buffer becomes full, the order is automatically switched to the next buffer area on that group.

Example edge cases:

  • You have a buffer that can fit 4 resources and 3 resources are already unloaded to the buffer. 1 resource is taken manually away from the buffer. Now Navithor still has info that there are 3 resources, even though 1 is carried away. Then Navithor will only deliver one resource to the buffer and set the buffer as full, even though in reality there would be space for another resource.

  • Resources are cleared on the buffer symbolic point, even though the area is actually full. Navithor creates order to take pallet to the buffer. When AGV comes to the beginning of the buffer it detects that the area is actually full and send the status to Navithor. Then Navithor changes the status to full, abort the drive to this buffer and select another available buffer if it is allowed. If selecting another buffer is not possible for example if the symbolic points are not Grouped, the AGV will just wait on that location until the buffer is cleared manually.

Instructions how to set Navitrol parameters

Required parameters

I,target_detection_type_selection, 3 # Enables ref docking or pallet detection, 0 = disabled, 1 = refl docking, 2 = pallet detection, 3 = pallet detection ver 2 (allows buffer detection)

I,target_detection_scan_selection, 3 # Defines which scanners are used for target detection: 1 = rear scanner only, 2 = front scanner only, 3 = both scanners (supported only in refl docking and pallet detection ver 2)

Pallet model should be set into the parameters with array index of 0. Important parameters for buffers are:

I,pallet_model_enabled[0], 1 # 1 = pallet model enabled

F,pallet_model_width_y[0], 0.80 # Used for determining width of the buffer to check if it is free or not Parameters that can be added to params.txt to enable this buffer feature, change how close to the object the AGV stops in the buffer zone and how early it starts checking, if the buffer is free or not:

I,buffer_lanes_enabled, 1 # 0 = disable buffer lane feature, 1 = enable.

F,buffer_stop_dist_from_object_when_unloading, -2.0 # Stopping distance (m) from the closest object, when unloading in a buffer. Calculated from navigation point to the first scanner measurements seen on the buffer zone.

F,buffer_check_start_dist_offset_m, 1.0 # The minimum distance from navigation point to the buffer zone, when the AGV starts checking if the unload buffer is free or not

F, buffer_stop_dist_from_object_when_loading, -0.05 # Stopping offset (m) for load buffer. Calculated from navigation point. Depends on the sensor, if this parameter needs to be set. If pallet is detected really early it might be useful to have some value to make sure that the pallet is actually loaded.

I, load_buffer_mode, 2 # 0 = none, 1 = detect any objects based on scanner measurements, 2 = detect a pallet, 3 = external sensor (Load Detected bit on Supervisor status or CAN)

I, unload_buffer_mode, 1 # 0 = none, 1 = detect any objects based on scanner measurements

Increasing speed on buffer lanes

Navitrol has implementation, that disables the safety scanner field based on distance from a detected object. This is useful if the AGV has safety scanner installed statically to the body of the AGV, so the scanner doesn't move with the fork.

The main goal for this implementation is to drive faster out from an unload buffer and faster in to a load buffer. If the scanner is safety scanner and its safety has to be disabled when the fork is down, because otherwise it will trigger safety stop. On the buffer lanes, the safety is disabled only when the AGV is close to the load or unload, so the AGV can drive faster and safer further away on the buffer lane.

Driving faster on buffer lanes with this kind of fork setup, requires Navitrol to handle the disable safety message more efficiently based on the distance to the pallet. This means:

On load buffer:

  • The route has a fork height above scanner, so the scanner can see where the pallet is on the buffer.

  • The route speed can be set to normal driving speed.

  • First, the scanner detects the pallet on the route.

  • The AGV starts slowing down to disable_safety_limit_speed when it reaches the load_buffer_slow_down_distance from the detected pallet.

  • When AGV reaches the distance of disable_safety_distance_on_buffer to the pallet, the safety is requested to be disabled (supervisor status section 11 Disable safety) and the fork is decreased to actuator_min_val.

    • Note! If pallet detection scanner can continue detecting pallet when the fork is on the floor, set this parameter to 1: buffer_allow_drive_end_update_when_safety_is_disabled

  • AGV drives under the pallet and picks it up.

  • Continues to unload location.

On Unload buffer:

  • The route has a fork height above scanner, so the scanner can see where the last unloaded pallet is on the buffer.

  • The route speed can be set to normal driving speed.

  • First, the scanner detects the pallet on the route.

  • The AGV starts slowing down to disable_safety_limit_speed when it reaches the unload_buffer_slow_down_distance from the detected pallet.

  • When AGV reaches the distance of disable_safety_distance_on_buffer to the pallet, the safety is requested to be disabled (supervisor status section 11 Disable safety).

  • The AGV drives to Unload location and unloads the pallet.

  • When AGV start driving to next location, it keeps the fork at actuator_min_val for the docking_limit_leave_speed_and_actuator_height_dist distance from the last unload location.

  • The speed is limited to docking_limit_leave_speed until the AGV is further away than this distance parameter or until the fork is above the actuator_disable_safety_max_value_mm.

  • The safety is requested to be disabled until the AGV reaches the disable_safety_distance_on_buffer distance from the last unload location.

  • When AGV is not anymore on this docking_limit distance, the speed and actuator height is taken from the route.

There is new supervisor section message to send the PLC information when Navitrol want to disable the safety on the scanner that is below the fork. The cases when Navitrol will send this message are:

  • When fork is decreased too low (the height can be set with parameter)

  • When the AGV is close to the load or unload location (the distance can be set with parameter). Note: the distance is calculated between the AGV and the detected pallet on buffer lane, not the symbolic point.

Setting up disable safety feature

  • When setting up the disable safety feature, the disable_safety parameter needs to be set to 1. If this parameter is 0, the supervisor message is not sent.

  • Then the disable_safety_distance_on_buffer parameter needs to be set to value, where the safety is disabled. This distance is calculated from AGV navigation point to the unload or load location (location where the navigation point is when the load or unload is done). This parameter should be larger than the safety fields on the safety scanner and it is used to disable the safety just before we arrive to load or unload point.

  • The actuator_disable_safety_max_value_mm parameter should have a value where the fork is just above the safety field and does not trigger the warning zones.

  • If the AGV has some limited speed when the safety is disabled, disable_safety_limit_speed parameter can be set to make sure that the AGV doesn't drive too fast during the limited safety.

Disable safety supervisor message

This message is sent from Navitrol to supervisor on the supervisor status field. This section (ID 11) needs to be requested by the supervisor. See Ethernet Communication specification between Navitrol and AGV supervisor_v4 document, how to request a message from Navitrol.
