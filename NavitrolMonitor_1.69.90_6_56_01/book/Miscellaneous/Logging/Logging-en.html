Overview

What is a log?

A Navitrol log is a file containing the history of most important data processed by Navitrol.

The purpose of a log is to store data so that possible issues can be easily located and solved.

Typically, logs are taken by the user when a problem is detected and sent to Navitec for further inspection to locate and solve the problem. However, advanced users can utilize the logs by themselves to solve simple problems.

What data a log contains?

With default settings a log contains around one minute's worth of history data -- the oldest data is constantly overwritten.

Logged data can be adjusted with Navitrol parameters, see section How to adjust logging -- parameters

Examples of data logged by Navitrol:

  • Position of the AGV

  • Inputs from and outputs to connected devices and software, such as

    • Scanners

    • Motor controllers and encoders

    • Supervisor / PLC

    • Fleet control

  • The state of Navitrol and its threads

  • Errors and warnings

Where are the logs stored?

By default, the logs are stored into "/home/navitec/logs"-folder of the IPC that is running Navitrol.

File system of the IPC can be accessed from Windows with programs, such as Samba or WinSCP, see section How to retrieve a log from the IPC.

The logs can also be downloaded when taking them using Navitrol Monitor or Navithor Client.

Log types

An overview to log types is given in this section. For instructions on how to a take log, see section.

There are two main log types: text logs and binary logs.

Text logs

There are three different types of text logs:

  • Basic Log

  • Start Log

Basic log - log.txt

A text log contains the basic runtime data as described in section What data a log contains?.

It is used in problem solving to see minor details about inner functions of Navitrol.

The data is readable by humans -- written in English.

The text log file names end with ".txt" and can be opened with any text editor, such as notepad++.

"log.txt" is included in most binary log packages excluding Continuous and Monitor's visual logs. Optionally it can be taken separately using Linux console commands.

Start log -- startlog.txt

A start log contains the same data as the basic log but is saved automatically to the log folder on the IPC after 5 seconds of Navitrol start.

A start log can be used to identify start up issues, such as invalid parameters in parameter file, and to see initialization data.

It is always named "startlog.txt" and previous start log will be overwritten when the program is started again.

Long log -- longlog.txt

A text log containing only data logged as high priority, such as position of an AGV and active errors. The log covers a longer period of time compared to the basic log but is much less detailed.

A long log can be used to see issues over long period of time. However, in most cases it does not contain enough details to solve those issues. Long logs are named "longlogXX.txt" where XX is running number: 01, 02.....

Binary logs

Navitrol binary logs contain data that go through the positioning and navigation, including laser scanner measurements and the environment (also known as map).

They can be used to see an overview of problematic situation, such as an area with low position confidence.

Binary logs are in binary format which means that they can be opened with Navitrol Monitor for a log playback by dragging and dropping a zipped file into Monitor window.

Zipped binary log file names end with ".gz".

Naming

  • Log name explained

    • log_<type><number/index>_<year.month.day>_<hour.minute.second>_<AGV position x coordinate>_<AGV position y coordinate>_<Navitrol version>.gz

    • Note that AGV position coordinates are only added when they are available.

  • Examples

    • log_remote_mon0091_2020.10.15_13.36.36_1.0_6.0_6.29.45.gz

    • log_con0075_2020.10.13_09.14.55_6.29.43.gz

    • log_cont_2020.10.21_10.51.22_192.168.56.101_-2,9_6,0.gz

Note: Binary logs are zipped packages that contain basic log and start log text logs. The text logs can be extracted by unzipping the ".gz"-file twice.

Monitor log -- log_remote_mon

A binary log taken using Navitrol Monitor.

Example:

log_remote_mon0091_2020.10.15_13.36.36_1.0_6.0_6.29.45.gz

Console log -- log_con/log_cons

A binary log taken using Linux console commands.

Example:

log_con0075_2020.10.13_09.14.55_6.29.43.gz (Navitrol version < 6.34)

log_cons0075_2020.10.13_09.14.55_6.29.43.gz (Navitrol versions 6.34 and later)

External log - log_ext

A binary log taken using external command, such as Supervisor interface

To adjust see the related parameters.

Example:

log_ext0001_2020.10.15_17.36.02_1.0_6.0_6.29.45.gz

Error log -- log_err

A binary log is taken automatically when an error is activated.

By default, only predefined errors are logged automatically. However, automatic logging for other general errors can be enabled with parameters.

Error logs are stored to separate error log folders under the log folder.

To adjust see the related parameters.

Example:

log_err_401_ERROR_VEH_INTE_HIGH_DT_1143_2020.10.15_11.56.29_-2.7_6.0_6.29.45.gz

Continuous / Long visual log -- log_cont

Continuous logs can be taken with Navitrol Monitor and depending on the selection they are saved either to Navitrol PC or to PC running the Navitrol Monitor application (see section Advanced Monitor logging).

Continuous logging to PC running Navitrol Monitor must be separately started from "Save and download logs"-window of Navitrol Monitor.

These logs can be activated to capture problems that happen rarely, and user cannot react in time to take a regular log that includes the incident.

To adjust logs saved to Navitrol PC, see the related parameters.

Note: does not include text logs.

Example:

log_cont_2020.10.21_10.51.22_192.168.56.101_-2,9_6,0.gz

Monitor's visual log -- log_monitor

This log can be saved with Navitrol Monitor and contains the currently saved binary log data in Navitrol Monitor's memory that can be played back (see more in section Advanced Monitor logging).

Hence, this log can be saved even if the connection to the AGV is lost.

Note: does not include text logs.

Example:

log_monitor0001_2023.10.20_15.05.08.47_6.49.02.gz

Thread monitor log -- log_mon

A binary log taken in case of an automatic restart.

Thread monitor is a part of Navitrol that restarts Navitrol in case of an unexpected shutdown.

To adjust see the related parameters.

Example:

log_mon0037_2020.05.14_12.39.47_0.0_0.0_6.27.25.gz

FSTOP log -- log_fstop

A binary log automatically taken in case of state change triggered to FSTOP for any reason.

This feature can be enabled with the parameter log_dump_on_fstop_max_files (max number of logs stored; defaults to 0).

Example:

log_fstop0007_2020.05.14_12.39.47_0.0_0.0_6.27.25.gz

FSTOP on ESTOP log -- log_estop

A binary log automatically taken in case of state change triggered to FSTOP on ESTOP.

This feature can be enabled with the parameter log_dump_on_estop_max_files (max number of logs stored; defaults to 0). If FSTOP logging using parameter log_dump_on_fstop_max_files is enabled then this log is not stored.

Also parameter pc_fstop_on_estop should be set (defaults to 1).

Example:

log_estop0007_2020.05.14_12.39.47_0.0_0.0_6.27.25.gz

Other

Navigation log -- navigationlog.txt

A separate text log named "navigationlog.txt" containing only times and related positions.

Compatible only with "positioning only"-AGVs. (Navitrol does not control the AGV).

Can be enabled with the parameter log_to_navigationlog Misc parameters.

Battery log -- battery_log.txt

A separate text log named "battery_log.txt" containing only battery related information.

It can be enabled and adjusted with parameters.

Core dump -- core

A separate log named "core" that can be included in the zipped Thread monitor log. If you encounter this log type, please forward it to the Navitec support.

How to take a log

Navitrol Monitor

Step by step instructions

  1. Start Navitrol Monitor by running NavitrolMonitor.exe

  1. Log in by entering the password (default: ntnavitec)

  2. Connect to Navitrol running on the AGV

  • Select an AGV and IP from the dropdown list in top left corner.
    • If the dropdown list is empty. Click "Edit"-button next to it and add new machine name and IP-address.
  • Click "Connect"-button
  1. Click "Save Logs"-button to open "Save and download logs"-window.

  1. Take a log by pressing "Save Log" button.

  1. Wait for the "Download Log Files" view and "Last saved: " text to be updated.

The log is now saved to the IPC and it can be downloaded using the "Download" button.

Advanced Navitrol Monitor logging

For advanced options click the "Advanced" field.

This expands the view of the form for additional logging options.

  1. Additional logs
    • Tick the checkbox for saving long visual logs (only binary) in Navitrol PC. When taking the logs with "Save log" button, log_cont*.gz logs (one or more) will be saved along with the regular log_remote_mon*.gz log. This can be useful when the log could not be saved immediately after the incident.
    • Tick the checkbox for saving Monitor's visual logs (only binary) in the PC running the Navitrol Monitor application. Click the "Browse.." button before saving the log with the "Save log" button to determine the location to store the log_monitor*.gz log. This log can be saved even if the connection to the AGV is already lost.
  2. Continuous logs
    • Tick the checkbox for saving continuosly long visual logs (only binary) to a folder the user can define clicking the "Browse.." button. This can be useful when the issue happens rarely. Remember to untick the checkbox before trying to playback the log_cont*.gz logs. Note also that Navitrol Monitor needs to be all the time connected to the AGV for saving the logs continuously.
  3. SSH Credentials
    • Click the "SSH Credentials.." button to determine the username and password for Navitrol PC connection. The credentials are needed for downloading / deleting the logs from the Navitrol PC.

Linux console

Instructions:

  1. Open Linux console on the AGV running Navitrol -Usually done using PuTTY connection - Default user: "root" and password: "navitrol"

    • Another option is to directly connect a screen and keyboard to the IPC

  2. Select log type to dump

  • Binary log
    • Log command: ./Navitrol dump
  • Text log
    • Log command: ./Navitrol dumpf

  1. Type the log command of selected log type and press enter

    • See previous step for available commands.
    • Example:

  2. Wait for the line telling that logging is finished.

The log is now saved to the IPC and can be downloaded using the "Download" button

Advanced console logging:

It is possible to filter the data written to the log file and filter and print log lines to the console while Navitrol is running. For further instructions, see section Advanced logging.

Supervisor interface

If supervisor interface is used, external-log---log_ext can be taken by sending message 1007. See "Ethernet Communication specification between Navitrol and AGV supervisor_v4.docx"-doument for details.

Navithor client

When taking a log with Navitrol client it is possible to include binary logs (monitor-log-log_remote_mon) from individual AGVs. Further instructions can be found from Navithor documents.

Basic steps:

  1. Open Navithor client

  2. Click "Save logs"-button on lower left corner to open "Log saver"-window.

  1. Select the AGV (or AGVs) from upper left section. (Navitrol logs are included only for selected AGVs)

  1. Click "Save the log files"-button in the bottom of the window.

  2. Wait for pop-up window informing that saving the log was successful. The log is now saved to the Windows system into the folder shown in the pop up window.

Note: By default, the log is saved to the Navithor Server's folder, but the pop-up window gives an option to download the files to the computer running Navithor client. See document "Navithor System - overview and Client user instructions " for further details.

How to retrieve a log from the IPC

The log files can be retrieved from the IPC running Navitrol in multiple different ways. This section describes the most common ones.

WinSCP

  1. Open WinSCP
  2. Connect to the AGV
  3. Navigate to the log folder
  4. Copy or move the files where wanted

Navitrol Monitor

  1. Open the "Save and download logs"-window from the Navitrol Monitor clicking "Save Logs" button.

  2. Click the "Refresh"-button located in lower left corner.

  1. Select the logs you want to download and click the "Download"-button

  1. Select the file location by following the instructions of the opened pop-up window.

File explorer / samba

  1. Open Windows file explorer
  2. Type the AGVs IP to the top bar: <IP>/home/Navitec
  3. Copy or move the files where wanted

Note: Some Windows settings and versions may prevent remote access using this method.

How to read a log

Reading a text log

A text log can be opened with text editors, such as notepad++.

What you can do with a text log:

  1. See the main operations and diagnostics of the AGV over a longer period of time (compared to binary logs)
  2. See XY coordinates and heading value
  3. See Received and performed drive orders
  4. See errors and warnings that are active or have been set / reset
  5. Analyze and troubleshoot low level information, such as:
  • Received and sent data values between Navitrol and other systems or sensors
  • Communication intervals
  • Values and information from active Navitrol features and functions

Structure of the text log -- Start of the file:

  1. When the log was taken
    • Date and time
  2. Navitrol build information
    • Navitrol version
    • Build time
    • Machine type
  3. Kernel version
  4. Enabled license options
  5. Parameters from the params.txt
    • Data type, name, value # comment
  6. Parameters from the memory (not in the image)
    • Located just after "Parameters from the params.txt"-section

Structure of the text log -- A log line:

  1. "Time"
    • Counted from the start of Navitrol
  2. "Thread : level"
    • Thread
      • From which thread the log line is coming from
      • See List of log threads
    • Level
      • Priority of the information
      • The smaller the number the higher the priority
  3. "Logged data"
    • Typically starts with a description and ends with a list of values

Binary log playback

Binary logs can be played back using Navitrol Monitor.

Note: Versions of Navitrol and Navitrol Monitor must match.

How to start playback:

  1. Start Navitrol Monitor
  2. Make sure that you are not connected to any AGV
    • To disconnect from an AGV press "Disconnect" button located next to "Save Logs" button.
  3. Open a zipped binary log file
    • Option 1 - Drag and drop the file to the map section of Monitor window
      • "The map section" is the grid with coordinates on the sides
    • Option 2 - Select file to open
      • Open "File"-menu from top left corner.
      • Select "Open log"-option
      • Follow the pop-ups

What can you do with playback:

  1. Review localization performance
    • Graphs and visualization of scanner measurements, environment points and reflectors
    • Ability to match scanner measurements to the environment points and reflectors
    • Amount of performed correction
  2. Review driving performance
    • Graphs and visualization of the position and heading of the AGV
    • Measured position and heading compared to the route and the stopping points
    • Measured speed values compared to the target values
  3. Review the quality of the environment
    • Position confidence graph values
    • Amount of visible environment points

How to use:

  1. Charts and data can be viewed as if an AGV was connected

  2. Log time can be scrolled by

    • Dragging the blue slider in the bottom middle
    • Or with arrow keys

  3. A time interval can be pinpointed by

    • Dragging the ends of the time slider

Helpful information that can be seen with the log playback:

  1. How well the scanner measurements match the environment
  2. Position confidence
  3. Warnings and errors
  4. Graphs of given references and received measurements

Tip: If Navitrol Monitor has been connected to an AGV and then disconnected, the last moments can be played back as it were a binary log.

How to adjust logging - Parameters

Logging can be adjusted with Navitrol parameters by modifying params.txt file and reloading the parameters. Most of these parameters are optional, which means that if they are not defined in the params.txt their default values are used. From Navitrol version 6.30.00 forward all logging parameters have default value of 0 and are all optional.

Note: Most of these parameters affect only the text logging as data of binary logging cannot be adjusted by the user.

Note: When changing parameters, check if the parameter is already defined before adding one. If a parameter is defined twice, the Navitrol gets parse error and will not work.

Amount of logging

Amount of logging to text files can be adjusted with parameters. Each logging parameter adjusts the logging of a single Navitrol feature.

Effects of high and low logging rates explained:

AttributeLowHighComment
Time period savedLongShortThe longer period of time the log covers, the easier it is to catch information about an issue into it, because the user has more time to take the log.
In worst cases the logs are filled in less than 5 seconds making it almost impossible to capture an error.
Amount of detailsLowHighThe higher the amount of logged details, the easier it is to solve an issue.
Solving some issues might be impossible to solve with minimal logging.
Very high logging rates can also cause data to be lost because of hardware limits.

As a rule of thumb, the amount of logging should be kept low unless there is reason to suspect problems with a certain feature. For example, supervisor logging should be increased when taking new messages to use and decreased when they are tested and working.

With logging parameters value 0 means that logging is disabled. Values above that increase the amount of logging step by step until the parameter specific max value is reached.

From Navitrol version 6.30.00 forward, all logging parameters are optional, and they are disabled by default.

All logging parameters of integer type can be set in params.txt followingly:

I,parameter_name, value

Example: I,log_veh_inte_eth, 1

While logging parameters of unsinged integer type should start with U:

Example: U, log_ext_target_det, 1

Note: If some feature is not used, changing its logging level will not have any effect.

Trouble shooting

WARNING_HIGH_LOGGING_RATE

WARNING_HIGH_LOGGING_RATE is a warning activated when logging rate of the text.log exceeds characters per second rate defined by the parameter logging_rate_warning_limit.

Too high logging rate can cause logged data to be lost or corrupted. It might also decrease the overall performance of the Navitrol.

It is recommended to fix this warning by how-to-adjust-logging---parameters.

Other option is to increase the value of the parameter. However, this approach only hides the problem instead of solving it.

Navitrol Monitor gives an error when opening binary log

Make sure that the Monitor version matches the version of the Navitrol which was running when the log was taken.

Navitrol Monitor does not accept a log file

Make sure that the file type is ".gz" in lower case. For example, "LOG_BIN.GZ" will not open but "LOG_BIN.gz" will.

Other problems

If you are facing a logging related problem and you cannot find an answer from this document, please contact Navitec support.

Advanced

Filtered console logging

It is possible to use filtered text logging with Linux console commands. The list of commands that can be used in order to write specific log messages to files and/or view them in real time are presented below. Amount of log lines and the threads of interest can be specified manually.

If [n] is 0 when using dumps or dumpl, log lines are displayed in real time as they are output by '.'.

Note: All available Navitrol console commands can be seen with ./Navitrol help -- Linux console command.

To the console screen

Dumping filtered data to console screen in `real time` can be helpful when running tests as often user is interested only in certain log lines and seeing them from the console screen is faster than taking a log and opening it separately.

Related commands:

'./Navitrol dumps' - echo last 100 lines of log on screen.
'./Navitrol dumps [n]' - echo last [n] lines of log on screen.
'./Navitrol dumps [n] [t]' - echo last [n] lines of log from thread [t] on screen.
'./Navitrol dumps [n] [t1] [t2]' - echo last [n] lines of log from threads [t1] and [t2] on screen.
'./Navitrol dumpl [n] "[string1]" "[string2]" "[...]"' - dump last n lines of log that contain any of the specified strings.
'./Navitrol dumpt [start] [end]' - dump lines between timestamps [start] and [end].

To a text log

Dumping filtered data to a text log is rarely useful as user can dump whole log and filter it with text editor.

Related commands:

'./Navitrol dumpf' - dump whole log to ./logs/log.txt.
'./Navitrol dumpf [n]' - dump last [n] lines of log to ./logs/log.txt.
'./Navitrol dumpf [n] [t]' - dump last [n] lines of log from thread [t] to ./logs/log.txt.
'./Navitrol dumpf [n] [t1] [t2]' - dump last [n] lines of log from threads [t1] and [t2] to ./logs/log.txt.
