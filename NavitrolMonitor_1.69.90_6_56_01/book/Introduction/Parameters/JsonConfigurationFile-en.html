JSON Configuration File

Updated 24.06.2024

Overview

There are two approaches to specify the configurations of an AGV. The primary and widely used method involves specifying the parameters within the params.txt file. However, as Navitrol's complexity grows, the need for a more structured and modular way has arisen. This necessity leads to devising a new approach by using the JSON file format. The JSON file format improves the readability and modularity of the configurations, but it currently supports only a limited set of parameters.

Setup

The path dedicated for the JSON configuration file is configuration/config.json. If this file is provided, its parameters will have a higher priority compared to params.txt, and override those parameters. Otherwise, if the file is not provided, a template JSON file will be generated in configuration/config.json.template with default values. The same as params.txt, JSON configuration file will be taken into account at the start or when the parameters are reloaded.

Structure

The structure of the JSON configuration file follows the standard format. It contains one main object, which is called the root object. Each object in the JSON file represents a module or a coherent abstract concept by a set of key-value pairs. The keys are of type string and the values can be Boolean, number, string, array, or another object. The order of the keys is not important, and the parameters correspond to the missing keys won't be reconfigured by the process of reading the JSON file. Also, the keys are case-sensitive, but whitespaces are ignored (except for the string values) which means arbitrary amount spaces, tabs, and newlines can be used to format the file. Moreover, in case of duplicate keys, the last key-value pair will be considered.

All objects share the comment key, which can be used to provide a description of the object or any comment that is relevant to the object. The comment key does not affect the behavior of the system and is ignored. An example of the JSON configuration file is shown below:

{
  "comment": "This is an example of the JSON configuration file",

  "Error Configuration": {
    "comment": "This object contains the error configurations",
    "DisableDefaultAutomaticLogging": false,
    "TakeAutomaticNavitrolLog": [666, 878],
    "DisableAutomaticNavitrolLog": [123],
    "Request3DPositioningBinaryLog": [878]
    "IncreaseToError": [123],
    "IncreaseToFatalError": [223],
  }
}

As mentioned earlier, the JSON configuration file provides a hierarchical modular structure to define the configuration. The following sections describe the supported sub-entities of the root object.

Error Configuration

Using Error Configuration object, we can override the default error configurations. The following keys are supported:

  • DisableDefaultAutomaticLogging: A Boolean value to disable the default automatic logging of errors. The default value is false.
  • TakeAutomaticNavitrolLog: An array of error IDs to trigger automatic Navitrol logging. If an error corresponding to one of those IDs is raised, the logging process will automatically be triggered. The default value is an empty array.
  • DisableAutomaticNavitrolLog: An array of error IDs to prevent automatic Navitrol logging. The logging process will not be triggered for those error IDs. The default value is an empty array.
  • Request3DPositioningBinaryLog: An array of error IDs to provide 3D positioning binary data in their log. If any of those errors is raised, in the logging process, binary log related to 3D positioning will also be collected. The default value is an empty array.
  • IncreaseToError: An array of warning IDs to override their severity level to error. The default value is an empty array.
  • IncreaseToFatalError: An array of error/warning IDs to override their severity level to fatal. The default value is an empty array.

By default, logging process will be triggered automatically for some error IDs. You can set DisableDefaultAutomaticLogging to true, in order to disable the default automatic logging. To explicitly specify the error IDs that should trigger logging or prevent it, the TakeAutomaticNavitrolLog or DisableAutomaticNavitrolLog key can be used, respectively. The behavior specified by those two parameters overrides the default behavior. In other words, if some error ID is specified in TakeAutomaticNavitrolLog list, logging process will be triggered for that error, irrespective of the default behavior for that error ID or the value of DisableDefaultAutomaticLogging. The same is true for DisableAutomaticNavitrolLog parameter. The priority of TakeAutomaticNavitrolLog is higher for the error IDs that are presented in both lists. Moreover, To collect 3D positioning binary data in the log for specific error IDs, you can use Request3DPositioningBinaryLog key.

If you want to increase the severity level of some warning/error IDs to error or fatal, the IncreaseToError or IncreaseToFatalError keys can be used, respectively. It is worth mentioning that the severity level of errors won't be decreased.

VDA5050

The VDA5050 object can be used to configure the VDA5050 interface. Action criticality is the only configurable feature currently. It can be configured like this:

{  
  "VDA5050": {
    "actions": {
      "pick": {
        "critical": false
      }
    }
  }
}

This example JSON would configure the pick action to not be considered critical.

Troubleshooting

If the desired configuration is not achieved, please make sure that the JSON configuration file is in the correct path (i.e., configuration/config.json), and the file is in the correct format. If the JSON configuration file is not provided, a template JSON file will be generated in configuration/config.json.template with default values. If the JSON configuration file is provided, but the file is not in the correct format, the ERROR_PARAM_PARSE error will be raised. If the JSON configuration file is loaded successfully, it overrides the related parameters in the params.txt file.

Additionally, the other reason that some configurations might not be applied, is the presence of duplicate keys in the JSON configuration file; in this case, the last key-value pair will be considered.

To diagnose an issue, you can check the Navitrol log file. Log lines with JSON Configuration prefix are related to the JSON configuration file.
