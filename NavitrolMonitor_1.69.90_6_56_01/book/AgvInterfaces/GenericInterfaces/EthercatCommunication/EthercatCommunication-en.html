NAVITROL USER INSTRUCTIONS - Navitec Systems Oy CONFIDENTIAL

EtherCAT Communication

Updated 15.03.2024

Overview

Introduction

EtherCAT is one of the control interfaces supported by Navitrol. It can be used for controlling different devices e.g. motor controllers, encoders, etc.

EtherCAT is a high-performance, low-cost, easy to use Industrial Ethernet communication technology with flexible topology. It is used for real-time master-slave communication between controllers and field devices.

For the AGV system an IPC with Navitrol is the master of the bus and all the field devices (motor controllers, encoders e.g.) are slaves.

Requirements

  • Not all the EtherCAT devices are supported by Navitrol. Supported devices list should be consulted to check if some specific device is supported.
  • At least one of the Ethernet port of the IPC should be reserved for EtherCAT communication
  • EtherCAT interface is supported by the Navitrol starting from version 6.40.00

Limitations

  • At the moment, it is not possible to use CAN and EtherCAT together.
  • The feature is available only for position control systems.

Operational principles

EtherCAT bus topology

A Typical EtherCAT device has two or more communication ports. This allows to combine devices on a bus into different topologies.

EtherCAT provides great flexibility in terms of topologies. At least, these are supported:

  • Line
  • Tree
  • Star
  • Drop line
  • Ring

Also, a combination of different topologies (e.g. line + drop line) is supported, as it shown on the figure bellow.

Available topologies in EtherCAT.

From the reliability point of view it is recommended to use a ring topology since it allows cable redundancy. For instance, if some cable between two slaves is broken or cut, communication with all the devices would be still possible. Note that in such a case, two Ethernet ports of the master device will be used.

Cable redundancy in EtherCAT.

Cable selection

EtherCAT supports standard cables used for Ethernet connections. The following cable types are supported:

  • 100 BASE-TX (copper cable up to 100 m between 2 nodes). In practice, it is typically a standard patch-cord.
  • 100 BASE-FX (fiber optics).

Typically, EtherCAT devices are having RJ45 8p8c connector. However, some of the devices can have some alternative connectors (e.g. M12 or some proprietary connector. This should be noted when ordering cables for the EtherCAT bus.

Connection to the IPC

Standard IPC for Navitrol has two Ethernet ports (RJ45, 8p8c). Both of them can be used for EtherCAT communication.

Note that it is not recommended to connect EtherCAT bus through the switch (to bridge master and bus segment). It can cause issues and extra delays not only with EtherCAT communication, but also with Ethernet communication through this port. Therefore, it is recommended to connect EtherCAT devices directly to the IPC Ethernet port.

Addressing

In EtherCAT network, master automatically assign addresses. Therefore, there is no need for manual address configuration.

Navitrol (inside IPC) assign addresses the following rules:

  • Slave address starts from 1.
  • Slave address of a field device is done by physical position in the network
  • Slave address is auto-incremented.

Note: all slaves on the bus should have unique slave addresses.

In practice, it means that all EtherCAT slaves are addressed starting from master using the auto-increment procedure in ongoing direction. So the closest device on the bus to master will have address 1, the next device will have address 2 and so on.

Simple addressing example.

Advanced addressing

Addressing should be straight-forward in case if devices have only two Ethernet ports. In some cases, it might be hard to define ongoing direction where address will be auto-incremented. For instance, if several different topologies are mixed together.

In such a case, it can be done in two ways:

  1. Theoretical way.

    Every EtherCAT devices have up to 4 communication ports. Data in every device is processed in the following way: it is received from the previous device to the input port (port 0). After that, data for the device itself is inserted and the address is incremented. Then, if exists, data goes to the port 3 and all the devices connected to port 3 will be addresses. The same happens after that with the port 1 if exists. Finally, data goes to the output port and devices connected to it will be addressed.

    Advanced addressing example.

  2. Practical way

    This option will show position and the device type in the bus. The only limitation is that it will not distinguish two devices of the same type between each other.

    2.1 Power the devices and set up basic EtherCAT communication parameters.

    2.2 Connect EtherCAT bus to Navitrol and start Navitrol

    2.3 Wait for 20 seconds and take log

    2.4 Open the log and search "slave(s) found on the bus".

    It will list all the device types on the bus and print addresses.

Set up

The following steps should be done in process of commissioning:

  1. Physical connection

  2. Setting up the device

  3. Changing parameters

Physical connection

Wiring is the first step in establishing the EtherCAT bus. All devices are connected together with a standard Ethernet cables. Wiring is recommended to start from the IPC. Slave devices should be connected directly to those ports.

Additionally, make sure that devices on the bus are powered prior to the operation and wired according to the manufacturer's instructions.

Setting up the device

Some devices might require additional settings before start operating with Navitrol. For instance, motor controllers typically require some additional setting, such as PID controller tuning. For setting up such devices, refer to the device-specific document.

However, some of the devices are configured automatically by Navitrol when connected.

Basic setup of Navitrol

Parameter set depends on the project. Therefore, in this document, only generic parameters for establishing the communication will be described.

Depending on the topology, ethercat_mode parameter should be set. In case of ring topology when both ports of the IPC are utilized, master redundant mode should be selected. Additionally, ethercat_inte_redundant interface should be set in addition to ethercat_inte_primary. In other topologies (when only one port of the IPC is used), master single mode should be selected. Additionally, ethercat_inte_primary should be set.

The other parameters (e.g. slave addresses) can be found in the device-specific documents. For instance, kinematic type related parameters (specifying slave addresses of motor controllers) can be found in motor controller specific document (e.g. for Synapticon motor controller).

NameTypeDefaultInformation
ethercat_enabledInt0Enables EtherCAT communication
ethercat_modeInt0EtherCAT mode, 0 - Master single, 1 - Master redundant
ethercat_inte_primaryStringName of primary Ethernet interface used for EtherCAT communication
ethercat_inte_redundantStringName of redundant Ethernet interface used for EtherCAT communication
# Example parameters for ring topology

I,ethercat_enabled,                1 #
I,ethercat_mode,                   1 #
S,ethercat_inte_primary,        eth0 #
S,ethercat_inte_redundant,      eth1 #


# Example parameters for other topologies

I,ethercat_enabled,                1 #
I,ethercat_mode,                   0 #
S,ethercat_inte_primary,        eth0 #

Troubleshooting

Navitrol itself can detect most of the issues, such as:

  • some EtherCAT related configuration issues (e.g. parameter issue)
  • some device missing from the bus
  • a device is located on the wrong place on the bus.

It is advised in all the cases to check not only active errors, but also terminal. Typically, some important additional information is printed to the terminal in case of error.

Most common mistakes

EtherCAT start up fails

Typically, it can be explained by one of the following reasons:

  • Device is defined in parameter file but not connected to the bus
  • Device address defined in parameters is not correct. If some wrong device type is located in a position specified in the parameter file, it will also prevent EtherCAT from starting up
  • Device address is duplicated in the parameters.

In all the above-mentioned cases, it is recommended to open Linux terminal and check Navitrol prints from there. Bellow, there are the instructions on how to detect the problem and recover from it.

Device not connected to the bus

Active errors: ERROR_ETHERCAT_NO_COMMS, ERROR_ETHERCAT_HIGH_DT and ERROR_ETHERCAT_DEVICE_NOT_FOUND

Terminal error logging example:

EC: 1 Ethercat slave(s) not found. Registered slaves 2, devices on the bus 1. Not enough devices on the bus
EC: Slave 2 name SOMANET registered but not found on the bus

How to recover: Check if missing device is powered and connected to the bus

Incorrect device address

Active errors: ERROR_ETHERCAT_NO_COMMS, ERROR_ETHERCAT_HIGH_DT and ERROR_ETHERCAT_DEVICE_NOT_FOUND

Terminal error logging example:

EC: 1 Ethercat slave(s) not found. Registered slaves 2, devices on the bus 2.
EC: Slave 3 name SOMANET registered but not found on the bus

How to recover: Check parameters related to the EtherCAT address for mentioned device. Calculate its position on the bus (see "Addressing" chapter) and correct the parameter or position.

Wrong device at some address

Active errors: ERROR_ETHERCAT_NO_COMMS, ERROR_ETHERCAT_HIGH_DT and ERROR_ETHERCAT_DEVICE_NOT_FOUND

Terminal error logging example:

EC: CheckSlaves - slave 1 name mismatch, on the bus EK1100, registered SOMANET
EC: 1 Ethercat slave(s) not found. Registered slaves 2, devices on the bus 10.
EC: Slave 1 name SOMANET registered but not found on the bus

How to recover: Check parameters related to the EtherCAT address for mentioned devices. Calculate it's position on the bus (see "Addressing" chapter) and correct the parameter or position.

Slave address duplication

Active errors: ERROR_ETHERCAT_NO_COMMS, ERROR_ETHERCAT_HIGH_DT and ERROR_ETHERCAT_CONFIG

Terminal error logging example:

EC: Ethercat_RegisterSlaveInfo - attempt to double register slave address 1. Check parameters

How to recover: Check parameters related to the EtherCAT address. Several devices have the same address on the bus. Correct parameters.

Errors and warnings

Navitrol communication related errors are listed below, with additional troubleshooting information.

Note that ERROR_ETHERCAT_HIGH_DT and ERROR_ETHERCAT_NO_COMMS might be a cause of ERROR_ETHERCAT_CONFIG or ERROR_ETHERCAT_DEVICE_NOT_FOUND.

840 ERROR_ETHERCAT_NO_COMMS

Description: Navitrol has no connection to at least one EtherCAT slave (or no valid data has been received) for 100 ms.

Error level: Error

Error reason: Navitrol has not been able to initialize/configure EtherCAT properly, or slaves at run time are not working as expected.

How to recover:

  • Check that EtherCAT parameters (ethercat_mode, ethercat_inte_primary, ethercat_inte_redundant) are defined correctly in params.txt

  • Check that EtherCAT bus is physically connected to defined interface(s)

  • Check that all EtherCAT hardware and slaves are properly connected and powered

843 ERROR_ETHERCAT_CONFIG

Description: Error during Ethercat slave configuration.

Error level: Error

Error reason: Ethercat slave configuration failed. Ethercat HW and slaves might not be properly connected or powered.

How to recover: Check the log for more details. Contact Navitec support.

844 ERROR_ETHERCAT_HIGH_DT

Description: Update interval from vehicle interface to EtherCAT is too high (over control_interval_us * control_interval_error_ratio)

Error level: Error

Error reason: Vehicle interface is not responding to EtherCAT. In most cases, this error is caused by some other Ethercat error or issue with initialization of AGV. Might be caused by CPU overload of the IPC.

How to recover: Check the log or terminal for more details. Contact Navitec support.

847 ERROR_ETHERCAT_DEVICE_NOT_FOUND

Description: At least one EtherCAT slave device is not found on the bus

Error level: Error

Error reason: Some of the devices expected by Navitrol are missing from the bus.

How to recover: Check the log or terminal for more details. Check that all devices are powered and connected to the bus. Check Navitrol parameters. Contact Navitec support.

Advanced troubleshooting

Navitrol always logs some amount of events related to EtherCAT communication and EtherCAT related issues. Additional information logging might be enabled by the parameters mentioned below. These settings increase the amount of logged data and should not be used without a reason. Increased logging level might be needed only in cases when some issues with communication are present.

EtherCAT advanced parameters

NameTypeDefaultInformation
log_ethercatInt0Log EtherCAT communication details, 0 - min level ... 3 - max amount of details

All the EtherCAT related logging can be found by searching "EC:" in the Navitrol text log.

The sequence for establishing communication with the EtherCAT devices is described below:

  1. Registration of the devices Navitrol reads parameter file and checks machine type to determine EtherCAT devices expected to be on the bus (device name, address, configurations etc.).

Logging example:

EC: Ethercat_RegisterSlave - slave 2 successfully registered, nbrODO 33, parseFuncExist 1, buildFuncExist 1
EC: Ethercat_RegisterSlave - slave 1 successfully registered, nbrODO 33, parseFuncExist 1, buildFuncExist 1
EC: Object registration finished
  1. Initialization of the interface (opening socket)

Navitrol reads basic EtherCAT parameters and initializes the communication on specified interfaces.

Logging example:

EC: ec_init on eth0 eth1 succeeded
  1. Slave checks:

Navitrol reads information from the bus and checks if bus configuration matches to the list of registered devices.

At least, the following checks are performed:

  • What number of devices is on the bus?
  • Is the number of registered devices corresponds to the number of devices on the bus?
  • Is registered device types are on correct positions?

Note that Navitrol is able to perform checks only for device types. It cannot distinguish two devices of the same type switched together on the bus. For instance, from the bus configuration, it cannot indicate that left and right motor controllers in diff drives are swapped together. It is only possible to detect that devices of the same type swapped together during the further steps in commissioning of the AGV (e.g. by test-driving)

Logging example:

EC: 2 slave(s) found on the bus
EC: On the bus - slave 1, name SOMANET, Vendor:product 22d2h:201h
EC: On the bus - slave 2, name SOMANET, Vendor:product 22d2h:201h
EC: Registered - slave 1, name SOMANET
EC: Registered - slave 2, name SOMANET
EC: CheckSlaves - slave 1 found, name SOMANET
EC: CheckSlaves - slave 2 found, name SOMANET
EC: All registered Ethercat slaves found successfully
  1. Configuring slave devices

Navitrol configures found devices (if needed) by sending SDOs to them.

Logging example:

EC: PRE_OP state reached for all slaves.
EC: SDO wr OK. Slave 2 idx 0x1016, subIdx 0x1, size 4, data 8C 00 00 00, dataOk 1, wkc 1
.....
EC: Slave configuration success
  1. Putting devices to operational state

Navitrol switches all the devices to the operational state prior to communication

Logging example:

EC: Request operational state for all slaves
OPERATIONAL state reached for all slaves
  1. Start communication

After successful start-up, Navitrol starts communication. Short after that, ERROR_ETHERCAT_NO_COMMS and ERROR_ETHERCAT_HIGH_DT should be reset.
