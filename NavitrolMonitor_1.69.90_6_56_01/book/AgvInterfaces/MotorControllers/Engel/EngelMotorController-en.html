Introduction

Engel motor controller is one of the motor controllers supported by Navitrol. Engel offers different range of single channel motor controllers that can be used for different purposes in the AGV.

In this documentation, instructions are provided for configuring the motor controllers to be interfaced with Navitrol.

Following motor controllers has been tested by Navitec Systems:

  • HFI series, HFI2660-N200-CO

However, according to documentation, other motor controllers also should be supported since CAN interface is common for all Engel motor controllers.

Operation of the device is possible either via fieldbus or via digital and analogue inputs and outputs. Communication via CANopen is supported by Navitrol.

The initial parameterisation/configuration of the device takes place with the parameterisation software DSerV.

For now, Engel motor controllers can be used:

  • For controlling steering in Schwarzmueller omni AGV

Electrical connections

Electrical connections can be divided into four groups:

  1. CAN bus. Terminal X2 with M12 connector is used to connect to the CAN bus. In order to connect to the can bus, pin 5 should be connected to CAN_L and pin 4 -- to CAN_H.

  2. Power supply. Terminal X1 (three internal wires) are used to connect to the power. Both main supply(A, BN) and logic supply (C, BK) should be powered.

  3. Series connection. Pins 10 (VT), 11 (GY-PK) and 8 (BK) should be connected to series adapter as it shown on the figure.

  4. Digital inputs and outputs.

The following digital inputs are used by Navitrol:

  1. DI1 (WH) is used in Engel state machine. It has to be powered, otherwise device cannot be put to "Operation Enabled" mode.

  2. DI3 (GN) is used as a negative limit switch for homing for steering.

  3. DI8 (BU-RD) can be used as an E-STOP (depending on Navitrol parameter I,mot_ctrl_engel_estop_enabled). If this pin is powered, safestop is disable. Once it loses power, safestop triggered (NC contact).

Commissioning steps

The following steps should be done in process of commissioning:

  1. Establishing serial connection to the motor controller

  2. Connection to DSerV software to enable CAN communication

  3. Changing Navitrol parameters

  4. Tuning internal PID controller

Note: By default, CAN communication is disabled in motor controller. It is mandatory to connect with DSerV software in order to enable it.

Establishing serial connection to the motor controller

Connection to the DSerV software is done using the serial port. Therefore, RS232-USB adapter should be used.

Connect wires as it shown in the Figure 1. Note that pins (2,3,5) shown on the figure are pinout of the DB9 adapter. In case of using Moxa adapter, pay attention that terminal block pin numbering is different to DB9 pinout. Additionally, it is important to connect ground (wire 8, BK) to pin 5 of the DB9.

Connection to DSerV software to enable CAN communication

In order to connect to motor controller, open DSerV software and select the COM-port to which motor controller is connected:

If wiring is correct and motor controller is powered, connection should be successfully established.

Once connection has been established, open tab Optimization -- Fieldbus operation. Select CANopen fieldbus. Enter bitrate (250 kbit/s is default for Navitrol) and Node-ID.

Note that all of the CANopen device on the bus should have different Node-IDs and the same bitrate.

Press "Send" and do a power cycle (remove power from the drive and apply it again).

Once it is done, motor controller is ready to communicate with Navitrol.

Changing Navitrol parameters

  • First, set Node ID to the corresponding parameter. For instance, if motor controller controlling steering in omni machine, set can_omni_steer_ctrl_nodeid to the same value that was set in previous step.

  • Set the communication parameters (can_ch and can_bitrate_k) according to CAN communication document.

  • Set some initial parameters for built-in PD controller mot_ctrl_engel_p_gain and mot_ctrl_engel_i_time_ms.

  • If motor controller is used for steering without external absolute encoder, homing offset can be changed using the parater mot_ctrl_engel_home_offset (4096 = 1 revolution). Corresponds to the object 607C in motor controller manual.

  • If estop should be enabled, set parameter mot_ctrl_engel_estop_enabled.

  • Other motor controller settings are done using SDOs by Navitrol.

Tuning PID coefficients

In order to tune PID controller, DSerV software in combination of Navitrol should be used. Note: tuning PID controller is typically done by customers.

In order to tune PID controller, step response should be given to the motor controller. Motor controller form can be used for giving out reference:

In motor controller form, the following features can be used:

  • Giving target angle

  • Giving target RPM

  • Brake release (should checked to drive the motor)

  • Safe stop (should be unchecked to drive the motor)

  • Reset fault

DSerV software capabilities

The following functions are available in DSerV software:

  1. Optimisation. This tab allows to change, save and view parameters inside the motor controller. Typically, all of the needed parameters are written by Navitrol, so there is no need to change anything.

  2. Monitor. This tab allows to see current values of engineering parameters, such as current, speed, temperature, power, voltage and so on. Used for monitoring purposes.

  3. Diagnostic. This tab allows to open two windows: CAN monitor and Ocsilloscope. Oscilloscope can be used for monitoring purposes and for setting up PID controller. CAN Monitor shows all of the CAN-objects and can be used for monitoring.

  4. Device information. Is shows state of the motor controller, state of the CAN bus, errors and information about fieldbus parameters.

Troubleshooting

CAN-bus errors

If CAN-bus is not working correctly, make sure that following things are checked:

  • CAN-bus has 120ohm resistors activated on the both end of the bus. The total CAN-bus impedance has to be approximately 60 ohms.

  • Check that all motor controllers and other CAN-devices in the bus have same baudrate

  • Make sure that all CAN-devices on the bus have individual NodeID

  • Double-check Navitrol parameters in this document

  • Make sure that you have done also CAN-bus grounding

    • If CAN-bus GND is not grounded, there might occur error frames in the CAN-bus

    • Higher baudrates require CAN-bus grounding because they require more accurate voltage levels

  • More information about CAN-bus related errors can be found from Navitrol - CAN TroubleShoot

Motor controller issues

If motor controller indicates some errors, try the following:

  • If motor controller is not rotated, check the console or find for "LogEngelErrors" in the log.

  • More information about error can be received from EMCY message, Engel should send in error state. To read EMCY, find in the console of log: "CAN: EMCY from node"

  • Try to establish serial connection to check errors directly from the motor controller

If it is written it the console "okToDrive 0", this means that homing process has not successfully finished yet. For the first 20 seconds, it is normally. If process lasts more than 20 seconds, it means that there are some issues with homing process (probably, limit switch cannot be found).

If it is written in the console "DI1 is not set. It is not possible to set motor controller to operation mode" then DI1 should be powered to solve the problem.
