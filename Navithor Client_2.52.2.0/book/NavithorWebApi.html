NAVITHOR USER INSTRUCTIONS

Web API

This document contains information on how to install and use services of the Navithor Web API.

Updated 04.10.2021

Settings

Settings are defined in WebApiSettings.json, it is located in the the ConfigFiles folder.

If it does not exist a default configuration is created when the server is started.

SettingDefaultValuesDescription
Enabledfalsetrue/falseEnables the web API server
Port4199any valid port numberPort of the web API server
BufferSize10241-65535Maximum frame size for the data buffer.
Securefalsetrue/falseEnables secure websockets. Must bind certificate to the port first.
Services{
"Example": "Basic",

ApiKeys{

}		List of valid ClientID:Api-Key pairsCredentials for connecting to the API when using Basic authentication.

Installation

Only installation step required is creating an URL registration for the user account. This can be done using netsh via commandline.

This is not required if NavithorServer is run as admin.

netsh http add urlacl url=http://+:4199/ user=everyone

Port number must be same as defined in the settings. User can be specified as a specific windows user. group "everyone" is operation language specific and might be different on a non english operating system.

More information:

https://docs.microsoft.com/en-us/windows-server/networking/technologies/netsh/netsh-http#add-urlacl

Basic authentication

If basic authentication is defined as the authentication method, the websocket request must contain the basic authorization header.

The username and password must match those defined in Settings -> ApiKeys.

It is recommended to setup secure websockets when using basic access authentication.

Authorization header field

from https://en.wikipedia.org/wiki/Basic_access_authentication

The Authorization header field is constructed as follows:

  1. The username and password are combined with a single colon (:). This means that the username itself cannot contain a colon.
  2. The resulting string is encoded into an octet sequence. The character set to use for this encoding is by default unspecified, as long as it is compatible with US-ASCII, but the server may suggest use of UTF-8 by sending the charset parameter.
  3. The resulting string is encoded using a variant of Base64 (+/ and with padding).
  4. The authorization method and a space (e.g. "Basic ") is then prepended to the encoded string.

For example, if the browser uses Aladdin as the username and open sesame as the password, then the field's value is the Base64 encoding of Aladdin:open sesame, or QWxhZGRpbjpvcGVuIHNlc2FtZQ==. Then the Authorization header field will appear as:

Authorization: Basic QWxhZGRpbjpvcGVuIHNlc2FtZQ==

Configuring secure websockets

To enable secure communication, you must specify https when registering the URL as in the installation step.

netsh http add urlacl url=https://+:4199/ user=everyone

Additionally you must bind a valid certificate to the port. You must specify the cert hash of a trusted certifcate inside the certificate store. This document does not provide means to create a certificate. If you wish yo use secure websockets you must provide a trusted certifcate by yourself. Appid can be anything, it can be used to identify the owning application.

netsh http add sslcert ipport=0.0.0.0:4199 certhash=0102030405060708090A0B0C0D0E0F1011121314 appid={00112233-4455-6677-8899-AABBCCDDEEFF}

More information. https://docs.microsoft.com/en-us/windows-server/networking/technologies/netsh/netsh-http#add-sslcert
